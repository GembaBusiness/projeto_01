## 1. Módulo 1: Identidade e Multi-Tenancy
Este é o núcleo do sistema, responsável por gerir as empresas (tenants), os utilizadores e a relação entre eles.

### 1.1. Entidades Principais
- **Companies (Empresas/Tenants):** É a entidade central da arquitetura. Cada registo representa uma empresa cliente, servindo como a âncora para o isolamento de dados.
- **Profiles (Perfis de Utilizador):** Representa os dados de um utilizador, estendendo a tabela auth.users do Supabase.
- **Memberships (Vínculos):** Tabela de junção que define a relação entre um user e uma company, permitindo que um utilizador pertença a várias empresas e que uma empresa tenha vários utilizadores.
- **Departments (Departamentos):** Armazena os departamentos específicos de cada empresa, permitindo uma segmentação interna.

### 1.2. Relacionamentos Chave
- A relação entre auth.users (utilizadores) e Companies é de muitos-para-muitos (N:M), materializada pela tabela Memberships.
- Uma Company pode ter muitos Departments (1:N), e um Membership pode estar associado a um Department.

## 2. Módulo 2: Autorização e Permissões
Este módulo define o que um utilizador pode fazer dentro de uma empresa, utilizando uma abordagem híbrida de RBAC (Role-Based Access Control) e ABAC (Attribute-Based Access Control).

### 2.1. Entidades Principais
- **Roles (Papéis):** Define um conjunto de permissões agrupadas (ex: "Admin da Empresa"). Podem ser globais ou customizados por uma company. Esta distinção é feita através da coluna company_id: se o valor for NULL, o papel é global; se tiver o ID de uma empresa, é um papel específico daquele tenant.
- **Permissions (Permissões):** Ações granulares no sistema (ex: projects.create).
- **Membership Roles:** Atribui um role a um membership específico.
- **Role Permissions:** Tabela de junção que conecta roles a permissions.

### 2.2. Relacionamentos Chave
- As permissões são contextuais: um role não é atribuído a um utilizador, mas sim a um membership (N:M), significando que as permissões de um utilizador dependem da empresa em que ele está a atuar.
- Um Role pode ter muitas Permissions (N:M), através da tabela role_permissions.

## 3. Módulo 3: Planos e Monetização
Este domínio controla o acesso a funcionalidades com base no plano comercial subscrito pela empresa.

### 3.1. Entidades Principais
- **Plans:** Define os pacotes comerciais (ex: Básico, Pro).
- **Features:** Catálogo de funcionalidades que podem ser controladas por plano (feature flags).
- **Prices:** Desacopla os planos dos seus preços (ex: mensal vs. anual).
- **Subscriptions:** Regista a subscrição ativa de uma company a um price.
- **Subscription History:** Funciona como um log de auditoria para cada subscrição (upgrades, cancelamentos, etc.).
- **Plan Features:** Tabela de junção que define quais features estão incluídas em cada plan.

### 3.2. Relacionamentos Chave
- Uma Company tem apenas uma Subscription ativa (1:1).
- Um Plan pode ter vários Prices (1:N) e várias Features (N:M através de plan_features).

## 4. Módulo 4: Interface e Navegação
Este domínio define a estrutura e o controlo de acesso aos elementos visuais da interface.

### 4.1. Entidades Principais
- **Navigation Items:** Armazena a estrutura hierárquica de todos os itens de navegação da aplicação (menus, submenus).
- **Nav Item Permissions:** Tabela de junção que define qual permission é necessária para visualizar um navigation_item.

### 4.2. Relacionamentos Chave
- O front-end pode consultar este módulo para renderizar dinamicamente a UI com base nas permissões do utilizador, que são derivadas dos seus roles.
- A relação entre Navigation Items e Permissions é de muitos-para-muitos (N:M).

## 5. Módulo 5: Conformidade e Auditoria
Este domínio gere o consentimento do utilizador para documentos legais e mantém uma trilha de auditoria.

### 5.1. Entidades Principais
- **Consent Types:** Catálogo versionado de documentos de consentimento (ex: Termos de Serviço v1.0, v1.1).
- **User Consents:** Registo de auditoria que vincula um user a uma versão específica de um consent_type que ele aceitou.

### 5.2. Relacionamentos Chave
- A relação entre Users e Consent Types é de muitos-para-muitos (N:M) através da tabela user_consents.

# 1.2 Modelo de Dados

Esta seção descreve as principais entidades de dados do sistema e como elas se relacionam. Para um detalhe completo das tabelas e colunas, consulte a seção [Base de Dados](../3-base-de-dados/).

## Entidades Principais

- **Profiles (Utilizadores)**: Representa um utilizador individual no sistema. Cada utilizador que se autentica tem um `profile`. Está ligado diretamente à tabela `auth.users` do Supabase.

- **Companies (Empresas/Organizações)**: Representa uma entidade de negócio ou uma organização. Um utilizador pode pertencer a uma ou mais empresas. Este é o nosso conceito de "tenant" principal.

- **Memberships (Vínculos)**: É a tabela de junção que define a relação entre um `profile` e uma `company`. Ela armazena também o papel (`role`) do utilizador dentro daquela empresa (ex: `admin`, `member`).

## Relacionamentos Chave

- Um `Profile` pode ter muitos `Memberships`.
- Uma `Company` pode ter muitos `Memberships`.
- Portanto, a relação entre `Profiles` e `Companies` é de muitos-para-muitos (N:M), através da tabela `Memberships`.
