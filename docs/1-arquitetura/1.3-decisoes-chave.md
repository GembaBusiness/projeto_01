# 1.3 Decisões Chave (Architecture Decision Records - ADRs)

Esta página regista as decisões arquiteturais mais significativas que foram tomadas durante o desenvolvimento do projeto. Usamos o formato de "Architecture Decision Records" (ADRs) para documentar o contexto, a decisão e as suas consequências.

---

## ADR-001: Escolha do Supabase como Backend-as-a-Service

- **Status**: Aceite
- **Data**: 2023-10-26

### Contexto

Precisávamos de uma solução de backend que nos permitisse acelerar o desenvolvimento, fornecendo autenticação, uma base de dados PostgreSQL e APIs de forma integrada. As opções consideradas foram:
1.  Desenvolver um backend customizado (e.g., com Node.js + Express).
2.  Usar o Firebase.
3.  Usar o Supabase.

### Decisão

Optamos por usar o **Supabase**.

### Consequências

**Positivas:**
- **Velocidade de Desenvolvimento**: Ganhámos uma enorme produtividade inicial, com autenticação, gestão de utilizadores e APIs de base de dados prontas a usar.
- **PostgreSQL**: Temos acesso a todo o poder do PostgreSQL, incluindo Row Level Security (RLS), que é fundamental para o nosso modelo multi-tenant.
- **Auto-documentação da API**: A API gerada automaticamente a partir do schema da base de dados reduz a necessidade de manter uma documentação de API básica manualmente.
- **Escalabilidade**: O Supabase oferece um caminho claro para escalar a infraestrutura.

**Negativas:**
- **Vendor Lock-in**: Embora o Supabase seja open-source, a nossa lógica de negócio (especialmente RLS e funções de base de dados) está fortemente acoplada à sua plataforma. Migrar para outra solução exigiria um esforço considerável.
- **Complexidade da Lógica na Base de Dados**: Para tirar o máximo partido do Supabase, muita lógica de negócio acaba por ser implementada em SQL (funções, triggers, RLS), o que pode ser mais difícil de testar e manter do que a lógica numa camada de aplicação tradicional.
