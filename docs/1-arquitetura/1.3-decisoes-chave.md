## 1. ADR-001: Adoção do Supabase como Plataforma de Backend-as-a-Service
- **Status:** Aceite
- **Data:** 2025-08-15

### 1.1. Contexto
Para a fundação do nosso sistema SaaS B2B, precisávamos de uma plataforma de backend que acelerasse o desenvolvimento, oferecendo serviços integrados de base de dados, autenticação, armazenamento e funções serverless, ao mesmo tempo que garantisse escalabilidade e segurança. As opções consideradas foram:
- **Opção A: Backend Monolítico Tradicional (ex: Node.js + PostgreSQL separado):** Controlo total sobre a arquitetura, mas com maior custo de desenvolvimento e manutenção da infraestrutura.
- **Opção B: Firebase/Firestore:** Solução madura e integrada, mas com uma base de dados NoSQL que poderia ser menos ideal para os nossos dados relacionais complexos.
- **Opção C: Supabase:** Plataforma open-source que utiliza PostgreSQL, oferecendo uma camada de serviços sobre uma tecnologia de base de dados robusta e familiar.

### 1.2. Decisão
Escolhemos o **Supabase**. O fator decisivo foi a combinação da produtividade de uma solução BaaS com o poder e a flexibilidade do PostgreSQL. Isso permite-nos usar funcionalidades avançadas como Row-Level Security (RLS) para implementar o multi-tenancy de forma segura e eficiente diretamente na base de dados.

### 1.3. Consequências
**Positivas:**
- Aceleração no desenvolvimento ao ter autenticação, DB e APIs prontas a usar.
- Segurança multi-tenant reforçada no nível da base de dados através de RLS.
- A base em PostgreSQL permite o uso de um ecossistema de ferramentas maduro.

**Negativas:**
- Curva de aprendizagem para dominar as especificidades do ecossistema Supabase.
- Dependência da continuidade e do roadmap de funcionalidades do Supabase.

## 2. ADR-002: Orquestração da Lógica de Negócio com Edge Functions e Funções de Base de Dados
- **Status:** Aceite
- **Data:** 2025-08-20

### 2.1. Contexto
Operações críticas, como o registo de uma nova empresa, envolvem múltiplos passos que precisam de ser atómicos e seguros. A lógica não poderia residir no cliente por razões de segurança.
- **Opção A: Lógica completa numa única Edge Function:** Aumentaria a complexidade e o risco de inconsistência de dados.
- **Opção B: Padrão Híbrido (Edge Function + Função de Base de Dados):** Usar uma Edge Function como orquestradora segura que invoca uma função PL/pgSQL na base de dados para executar a lógica de negócio de forma transacional.

### 2.2. Decisão
Adotámos o **Padrão Híbrido**. Uma Edge Function atua como ponto de entrada da API, e a lógica de negócio principal é encapsulada numa função de base de dados (PostgreSQL), que garante atomicidade. Se a função da base de dados falhar, a Edge Function é responsável pela ação de compensação (ex: apagar o utilizador criado no Auth).

### 2.3. Consequências
**Positivas:**
- Segurança máxima (chaves de administrador nunca saem do backend).
- Integridade de dados garantida através das transações atómicas do PostgreSQL.
- Separação clara de responsabilidades.

**Negativas:**
- Arquitetura mais complexa, envolvendo a gestão de código Deno/TypeScript e PL/pgSQL.
- Requer implementação cuidadosa da lógica de compensação para evitar estados inconsistentes.

## 3. ADR-003: Implementação de Autorização Híbrida (ABAC + RBAC)
- **Status:** Aceite
- **Data:** 2025-08-25

### 3.1. Contexto
O sistema de permissões precisa de garantir o isolamento de dados entre tenants e, ao mesmo tempo, gerir o que um utilizador pode fazer dentro da sua própria empresa.
- **Opção A: Apenas Role-Based Access Control (RBAC):** Simples, mas complexo para garantir o isolamento de tenants.
- **Opção B: Apenas Attribute-Based Access Control (ABAC):** Flexível, mas pode ser complexo de gerir e auditar.
- **Opção C: Abordagem Híbrida:** Usar ABAC para a regra fundamental de isolamento e RBAC para as permissões funcionais.

### 3.2. Decisão
Implementámos uma **Abordagem Híbrida**. A base da segurança é o ABAC, implementado via Row-Level Security (RLS) no PostgreSQL. Sobre essa camada, um sistema RBAC tradicional (roles, permissions) gere as permissões funcionais.

### 3.3. Consequências
**Positivas:**
- Defesa em profundidade com RLS como barreira de segurança no nível da base de dados.
- RBAC simplifica a gestão de permissões do dia-a-dia.
- Sistema altamente escalável.

**Negativas:**
- Requer conhecimento sólido tanto de RBAC como das políticas de RLS do PostgreSQL.
- A depuração de problemas de permissão pode ser mais complexa.

## 4. ADR-004: Adoção do WeWeb como Plataforma de Frontend
- **Status:** Aceite
- **Data:** 2025-09-02

### 4.1. Contexto
Para a UI, procurávamos uma solução que permitisse desenvolvimento rápido, boa integração com o Supabase e que não nos prendesse a um ecossistema proprietário.
- **Opção A: SPA Tradicional (Vue.js/React):** Controlo total, mas ciclo de desenvolvimento mais longo.
- **Opção B: Plataformas No-Code/Low-Code generalistas (Bubble):** Rápidas, mas com limitações de escalabilidade e customização.
- **Opção C: WeWeb:** Plataforma visual que gera código Vue.js standard, oferecendo um equilíbrio entre velocidade e flexibilidade.

### 4.2. Decisão
Escolhemos o **WeWeb**. A decisão baseou-se no seu equilíbrio ideal entre velocidade de desenvolvimento e flexibilidade técnica, permitindo construir visualmente e, ao mesmo tempo, recorrer a código Vue.js para funcionalidades complexas.

### 4.3. Consequências
**Positivas:**
- Redução drástica do tempo de desenvolvimento da UI.
- Menor dependência de uma equipa de frontend especializada.
- Mitigação do risco de vendor lock-in ao permitir a exportação do código Vue.js.

**Negativas:**
- Dependência da plataforma WeWeb (roadmap, preços).
- A depuração pode exigir conhecimento tanto da interface do WeWeb quanto do código Vue.js.
- A plataforma é focada em aplicações web, não sendo possível construir aplicações móveis nativas diretamente.
